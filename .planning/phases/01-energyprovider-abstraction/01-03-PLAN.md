<objective>
Verify that Plans 01 and 02 together produce a behavior-identical refactor — that is: with `ENERGY_DELEGATION_MODE=true` and no new env vars set, a USDT-TRC20 sweep through `transfer_trc20_from` performs the same RPC sequence, broadcasts the same kinds of transactions, and produces the same kinds of log lines as pre-refactor master.

Purpose: this is the Phase-1 acceptance gate. Plans 01 and 02 are mechanical lifts with high confidence, but the codebase has zero automated tests (per TESTING.md) and `transfer_trc20_from` is the highest-stakes path in the system (it touches private keys + on-chain assets). We do not ship the refactor without an explicit verification that the happy path still works.

This plan is intentionally conservative: it does NOT attempt to introduce a test suite (out of scope for Phase 1; per TESTING.md adopting `pytest` is a future, separate piece of work). It uses two lower-cost mechanisms:
1. **Structural verification** — a deterministic, offline check that the lifted code is shape-equivalent to the master closures. This is cheap, repeatable, and catches accidental logic edits.
2. **Live / fixture-replay smoke** — a single invocation of `transfer_trc20_from` (preferably against the operator's existing staging stack; alternatively against a mocked tronpy client) confirming the delegation+transfer+release sequence works end-to-end.

Output: a smoke-verification record at `.planning/phases/01-energyprovider-abstraction/01-03-SMOKE.md`.

Scope guardrail: this plan does NOT introduce a `tests/` directory, does NOT add `pytest` to requirements, does NOT modify any source code (it only reads `app/energy_provider.py` and `app/tasks.py`), and does NOT push to remote. The smoke run is a one-shot human-supervised invocation; if it fails, the failure is reported and Plans 01-02 are revised before merging.
</objective>
