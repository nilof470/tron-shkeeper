```python
class EnergyProvider(ABC):
    @abstractmethod
    def acquire(
        self,
        receiver: str,
        energy_to_provision: int,
        account_resource: dict,
        *,
        minimum_energy_required: int | None = None,
    ) -> bool: ...
    @abstractmethod
    def release(self, receiver: str) -> None: ...

class StakingEnergyProvider(EnergyProvider):
    # Concrete impl lifted from the closures.
    ...

def get_energy_provider() -> EnergyProvider:
    return StakingEnergyProvider()  # Phase 1 unconditional
```

Behavior contract (from Plan 01 spec):
- `acquire` internally computes `sun_needed` from `(energy_to_provision, account_resource)` using the same TotalEnergyWeight / TotalEnergyLimit ratio + `ENERGY_DELEGATION_MODE_ENERGY_DELEGATION_FACTOR` formula. The caller does NOT compute sun_needed any more — it passes the chain-side energy amount to provision and the account_resource snapshot, and the provider does the rest.
- `minimum_energy_required` preserves the original closure's outer-scope `energy_needed` recheck. In the additional-delegation path, pass `energy_diff` as `energy_to_provision` but pass the full transfer-wide `energy_needed` as `minimum_energy_required`, so `StakingEnergyProvider.acquire` rechecks `EnergyLimit >= energy_needed` after provisioning. Without this keyword, the top-up path would incorrectly recheck only `EnergyLimit >= energy_diff`.
- `acquire` returns True on success, False on terminal failure. Caller MUST `return` (without broadcasting) on False.
- `release` only does the celery dispatch (DEVMODE_CELERY_NODELAY check + .delay()/sync). It does NOT check `config.ENERGY_DELEGATION_MODE` — the CALLER does that, because in Phase 1 release is only called inside the existing `if config.ENERGY_DELEGATION_MODE:` block.
